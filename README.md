# Product & Marketing Analytics

![Tests](https://github.com/exist-ty/product-marketing-analytics/actions/workflows/test.yml/badge.svg)

Пет-проект уровня Data/Product Analyst: SQL-витрины и Python-визуализация поверх
staging-слоя, который готовит соседний ETL-репозиторий
[`etl-portfolio`](../etl-portfolio). Разделение осознанное: там — приём и
очистка сырых данных, здесь — их анализ и бизнес-интерпретация.

Три вопроса, на которые отвечает этот репозиторий:

1. Какие каналы привлечения окупаются (CAC/CPL/ROMI), а какие пора остановить?
2. Сколько в среднем приносит клиент к N-му заказу (LTV)?
3. Как быстро "отваливаются" клиенты после регистрации (cohort retention)?
4. Можно ли предсказать отток клиента по его поведению до того, как он ушёл?
5. Как выглядит тот же rollup-запрос в колоночном OLAP-движке (ClickHouse) —
   и где реально проходит граница, после которой он обгоняет Postgres?
6. Сколько реально стоит пересчёт витрины на каждый SELECT — и окупается ли
   материализация на этом объёме данных, а не только "в теории"?

## Стек

PostgreSQL (window functions, CTE, MATERIALIZED VIEW), ClickHouse (MergeTree,
AggregatingMergeTree), Python (pandas, matplotlib, scikit-learn, scipy),
Jupyter.

## Структура

- `sql/marts.sql` — три VIEW поверх `stg_customers` / `stg_orders` / `stg_marketing_spend`
- `sql/materialized_marts.sql` — те же три витрины как MATERIALIZED VIEW с
  уникальными индексами (см. «Материализованные витрины» ниже)
- `scripts/refresh_marts.py` — `REFRESH MATERIALIZED VIEW CONCURRENTLY` после
  прогона ETL, не на каждое чтение
- `scripts/compare_view_vs_materialized.py` — честный замер VIEW vs
  MATERIALIZED VIEW по всем трём витринам
- `notebooks/analysis.ipynb` — выполненный ноутбук с графиками и интерпретацией (открывается прямо на GitHub)
- `src/analytics/visualization.py` — тестируемая функция построения ROMI-графика
- `ml/build_features.py` — customer-level датасет для churn-классификации (label + фичи)
- `ml/train_churn_model.py` — Logistic Regression / Random Forest, ROC-AUC, PR-AUC, feature importance
- `clickhouse/schema.sql` — `order_events` (MergeTree) + инкрементальная
  витрина `channel_monthly_revenue` (AggregatingMergeTree + MATERIALIZED VIEW)
- `clickhouse/load_to_clickhouse.py` — реплицирует `stg_orders`/`stg_customers`
  из Postgres в ClickHouse (идемпотентно, TRUNCATE перед вставкой)
- `clickhouse/compare_engines.py` — честный замер: та же агрегация в Postgres
  VIEW-стиле vs ClickHouse (с нуля / через инкрементальную витрину)
- `sql/ab_test_schema.sql` — таблица `ab_test_email_campaign`
- `ab_test/generate_experiment.py` — симулирует рандомизированный A/B-тест
  на реальном пуле клиентов (см. «A/B-тестирование» ниже)
- `ab_test/analyze_experiment.py` — SRM-проверка, two-proportion z-test,
  доверительный интервал, power-анализ, ship/no-ship вердикт
- `ab_test/simulate_power_validation.py` — Monte-Carlo валидация формулы
  power-анализа независимой симуляцией (не пересчёт эксперимента на
  большей выборке ради красивого p-value, см. «A/B-тестирование» ниже)
- `tests/` — pytest для `visualization.py`, `ml/build_features.py`,
  `clickhouse/load_to_clickhouse.py` и `ab_test/`

## Как запустить

Требует запущенный и наполненный `etl-portfolio` (см. его README) — эта база
и есть источник данных.

```
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env   # указать пароль от того же etl_portfolio
psql -U postgres -d etl_portfolio -f sql/marts.sql
psql -U postgres -d etl_portfolio -f sql/materialized_marts.sql
psql -U postgres -d etl_portfolio -f sql/ab_test_schema.sql
jupyter nbconvert --to notebook --execute --inplace notebooks/analysis.ipynb
python ml/build_features.py
python ml/train_churn_model.py
python scripts/compare_view_vs_materialized.py
python ab_test/generate_experiment.py
python ab_test/analyze_experiment.py
python ab_test/simulate_power_validation.py
pytest
```

После каждого нового прогона ETL (не на каждое чтение дашборда):
```
python scripts/refresh_marts.py
```

Для ClickHouse-слоя дополнительно (после `docker compose up -d clickhouse`,
креды в `.env` — `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD`):

```
clickhouse-client --host localhost --multiquery < clickhouse/schema.sql
python clickhouse/load_to_clickhouse.py
python clickhouse/compare_engines.py
```

## Витрины (`sql/marts.sql`)

- **`mart_channel_economics`** — CAC, CPL, ROMI по каналу/месяцу + накопительный
  спенд/выручка через `SUM() OVER (PARTITION BY channel ORDER BY spend_month)`.
- **`mart_customer_ltv`** — накопительная выручка клиента и номер заказа через
  `ROW_NUMBER()` / `SUM() OVER (...ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)`,
  без единого `GROUP BY`.
- **`mart_cohort_retention`** — классический cohort-анализ: CTE считает размер
  когорты по месяцу регистрации, затем помесячную долю вернувшихся клиентов.

## Материализованные витрины (`sql/materialized_marts.sql`)

Все три VIEW выше пересчитываются целиком на каждый `SELECT` — CTE, JOIN'ы и
оконные функции выполняются заново, даже если данные не менялись с прошлого
запроса. ETL — батчевый пайплайн (данные меняются раз за прогон, не
непрерывно), а дашборд/API читают витрину много раз между прогонами — платить
полным пересчётом за каждое такое чтение бессмысленно. Рядом с каждым VIEW
добавлена версия `..._mv` (`MATERIALIZED VIEW`), которая хранит результат и
обновляется только явным `REFRESH` (`scripts/refresh_marts.py`, вызывается
после ETL, не на каждое чтение).

**Честный замер** (`scripts/compare_view_vs_materialized.py`, медиана из 20
прогонов на реальных данных этого проекта):

| витрина | VIEW | MATERIALIZED VIEW | ускорение |
|---|---|---|---|
| `mart_channel_economics` | ~5-9 ms | ~1-2 ms | ~5x |
| `mart_customer_ltv` | ~17 ms | ~9-11 ms | ~1.6-1.9x |
| `mart_cohort_retention` | ~10 ms | ~0.6-0.9 ms | ~12-16x |

В отличие от честного результата с ClickHouse (там Postgres выигрывал из-за
фиксированных накладных расходов другого движка на маленьком объёме), здесь
речь про один и тот же движок и одно и то же соединение — разница целиком
в том, что MATERIALIZED VIEW не пересчитывает JOIN'ы и оконные функции
заново, а просто читает готовую таблицу. `mart_cohort_retention` выигрывает
больше всех — у него самый тяжёлый план (CTE с `DISTINCT` и `GROUP BY` по
двум таблицам); `mart_customer_ltv` выигрывает меньше всех — оконная функция
там всего одна и не такая дорогая, разница по сути в накладных расходах
на JOIN.

**Плата за это — свежесть.** MATERIALIZED VIEW отдаёт данные на момент
последнего `REFRESH`, а не на момент запроса — если дашборд должен видеть
заказ, оформленный минуту назад, до следующего `REFRESH` он его не увидит.
Здесь это осознанный компромисс, потому что ETL и так батчевый: витрина не
обязана быть свежее, чем сами исходные данные. Для `REFRESH ...
CONCURRENTLY` (не блокирует чтение во время пересчёта) Postgres требует
хотя бы один `UNIQUE`-индекс на витрине — без него доступен только обычный
`REFRESH`, блокирующий `SELECT`ы на время пересчёта.

## Результаты на текущем датасете

Витрины посчитаны на реально сгенерированных и загруженных данных (200 клиентов,
~2000 заказов, 72 месяца×канала маркетингового спенда) — не придуманы вручную.

**ROMI по каналам за весь период** (доход того же месяца минус спенд, к спенду):

| channel | total spend | avg CPL | avg CAC | ROMI |
|---|---|---|---|---|
| referral | 639 | 11.6 | 23.7 | **+78.9** |
| seo | 3 007 | 15.1 | 68.3 | +22.0 |
| email | 647 | 5.8 | 38.1 | +19.4 |
| context_ads | 9 309 | 45.0 | 131.1 | +16.7 |
| social_ads | 7 540 | 33.5 | 183.9 | **+10.1** |

Вывод: `social_ads` и `context_ads` тянут больше всего бюджета при самом слабом
ROMI среди прибыльных каналов — первые кандидаты на пересмотр (см. график в
ноутбуке, где отдельные месяцы этих каналов уходят в отрицательный ROMI).

**В деньгах**: на каждый потраченный $1 `social_ads` и `context_ads` возвращают
$11.1 и $11.7 выручки того же месяца, `referral` — $79.9. Но это не значит
"просто перелить бюджет в referral" — абсолютный spend `referral` за весь
период (639) на порядок меньше, чем у `social_ads` (7 540): высокий ROMI
здесь считается на маленькой базе (сарафанное радио/реферальная программа
физически не масштабируется линейно с бюджетом), тогда как `social_ads` и
`context_ads` — каналы с доказанным объёмом, где решение "снизить спенд на
X%" реалистично оценить в деньгах уже сейчас, не дожидаясь более сложной
модели предельной отдачи (marginal ROMI) по каждому каналу.

**Кросс-репо наблюдение.** `referral` — лучший канал по ROMI здесь, но в
[`support-triage-llm`](../support-triage-llm) `channel_triage_summary.py`
показывает у `referral` и наибольшую долю негативных/high-priority обращений
в поддержку (25% против 10-12.5% у остальных каналов). На n=45 обращений это
не статистически значимый вывод (см. честную оговорку в том репозитории), но
направление стоит держать в голове: дешёвое привлечение через referral не
обязательно означает довольных клиентов — прежде чем масштабировать канал по
ROMI, честно было бы также посмотреть на качество полученных клиентов, а не
только на стоимость привлечения.

**LTV**: средняя накопительная выручка растёт с 508 у.е. к 1-му заказу до
~2 529 у.е. к 5-му (из 200 клиентов до 5-го заказа доходят 192) — повторные
заказы дают сопоставимую с первым заказом выручку, удержание так же важно,
как первичное привлечение.

**В деньгах**: даже у самого дорогого канала (`social_ads`, avg CAC 183.9)
выручка первого заказа (508) окупает CAC почти в 2.8 раза — то есть на этом
датасете привлечение окупается уже на первом заказе для всех пяти каналов, а
не только для дешёвого `referral` (CAC 23.7). Это меняет вопрос с "окупается
ли канал вообще" на "какой канал окупается быстрее" — и здесь `social_ads`
и `context_ads` всё равно проигрывают по скорости, просто не в убыток.

**Retention**: в среднем по когортам ~59% клиентов оформляют повторный заказ в
тот же месяц регистрации, к 1-му месяцу проседает до ~47%, дальше колеблется
55-71% без устойчивого тренда на этом объёме данных.

**В деньгах**: провал с 59% до 47% между месяцем регистрации и 1-м месяцем —
самое резкое падение на всей кривой (12 п.п., больше, чем любой последующий
шаг). Практический вывод: если в проекте появится ре-энгейджмент-кампания
(email/пуш через N дней после регистрации), максимальный ожидаемый ROI на
потраченный доллар именно здесь, а не в середине или в хвосте кривой,
потому что это точка наибольшего оттока в абсолютном выражении.

Полные графики (ROMI bar chart, LTV-кривая, cohort heatmap) и интерпретация —
в [`notebooks/analysis.ipynb`](notebooks/analysis.ipynb).

## Ограничение методологии

`revenue_same_month` в `mart_channel_economics` считает выручку только за тот
же календарный месяц, что и маркетинговый спенд — это занижает ROMI каналов с
более длинным циклом принятия решения (типично для B2B/дорогих товаров).
Более честный расчёт использовал бы окно атрибуции (например, 60/90 дней)
вместо строгого календарного месяца — следующий шаг для этой витрины.

## Предиктивный ML: churn prediction (`ml/`)

Мост от описательной аналитики (LTV, cohort retention) к предиктивной:
бинарная классификация оттока клиента.

**Label.** Явного события "отписки" в данных нет, поэтому отток определён
поведенчески: клиент считается оттёкшим, если последний заказ старше 90 дней
относительно `cutoff = max(order_date)` в датасете. Проверено: все клиенты
зарегистрированы минимум за 90 дней до cutoff — цензурирования по времени
наблюдения нет (не засчитываем "свежих" клиентов в отток нечестно).

**Признаки** (`ml/build_features.py`, один ряд на клиента): `total_orders`,
`total_revenue`, `avg_order_value`, `tenure_days`, `first_order_gap_days`,
`order_frequency`, `channel`. `days_since_last_order` в датасет входит, но
**не** используется как фича — это то же поле, из которого построен label
(иначе — target leakage).

**Обучение** (`ml/train_churn_model.py`): stratified train/test split (75/25),
`class_weight="balanced"` в обеих моделях — на 200 клиентах отток случается в
8.5% случаев (17 из 200), без балансировки классификатор просто предсказывал
бы "не ушёл" всегда и получал 91.5% accuracy при нулевой пользе.

| Модель | ROC-AUC | PR-AUC |
|---|---|---|
| Logistic Regression | 0.52 | 0.13 |
| Random Forest | 0.44 | 0.11 |

ROC-кривые и важность признаков (RF `feature_importances_`, LR-коэффициенты
на стандартизированных фичах):

![ROC curves](ml/plots/roc_curves.png)
![Random Forest feature importance](ml/plots/rf_feature_importance.png)
![Logistic Regression coefficients](ml/plots/lr_coefficients.png)

**Честный результат: AUC на уровне случайного угадывания — и это ожидаемо.**
`scripts/generate_data.py` (etl-portfolio) назначает `channel` клиенту и
частоту заказов случайно, без встроенного механизма "этот клиент похож на
тех, кто уходит" — в данных физически нет сигнала, который отличал бы
будущих отточных клиентов от активных. Модель, показывающая AUC≈0.5 на
данных без сигнала — это correct, не баг. Демонстрируется здесь не
"предсказательная сила на реальном бизнесе" (бизнеса нет, данные
синтетические), а сам workflow: honest train/test split, обработка
дисбаланса классов, набор метрик, интерпретация важности признаков — то,
что применяется к реальным данным без изменений. Дополнительное ограничение:
17 положительных примеров на весь датасет (4 в тестовой выборке) — при таком
n любая метрика шумит, доверительный интервал по ROC-AUC здесь шире, чем
разница между моделями.

**Бизнес-следствие честного AUC≈0.5**: на этих данных модель нельзя
подключать к реальной retention-кампании — таргетирование "топ-N клиентов по
churn-скору" при отсутствии сигнала не отличается от случайного выбора
клиентов, но создаёт иллюзию целенаправленности и тратит бюджет кампании
так же вслепую, как и без модели, только с дополнительными затратами на её
поддержку. Честная альтернатива на реальных данных — сначала статистически
проверить, что `channel`/поведенческие признаки вообще коррелируют с
оттоком, и только потом обучать модель под кампанию, а не наоборот.

## OLAP-слой: ClickHouse рядом с Postgres (`clickhouse/`)

Все витрины выше — Postgres VIEW: пересчитываются целиком на каждый SELECT.
Это нормально для этого объёма (~2000 заказов), но не масштабируется на
десятки/сотни миллионов строк. Рядом со staging-слоем поднят второй,
OLAP-профилированный движок на тех же исходных данных — не замена
Postgres-витринам, а демонстрация второй половины типичной data-платформы
(OLTP/staging + OLAP-аналитика).

**Схема (`clickhouse/schema.sql`).** `order_events` — `MergeTree`,
`PARTITION BY toYYYYMM(order_date)`, `ORDER BY (channel, order_date,
customer_id)`: физический порядок хранения на диске под access pattern
"фильтр/группировка по каналу и диапазону дат", тот же, что у
`mart_channel_economics`. Поверх — `channel_monthly_revenue`
(`AggregatingMergeTree`) и `MATERIALIZED VIEW ... TO`, который агрегирует
каждый вставляемый блок сразу при загрузке (`sumState`/`countState`), а не
при чтении: агрегат читается через `-Merge`-комбинаторы
(`sumMerge`/`countMerge`), которые лишь комбинируют уже посчитанные
состояния — принципиально другая модель пересчёта, чем у Postgres VIEW.

**Загрузка (`clickhouse/load_to_clickhouse.py`).** Реплицирует
`stg_orders`/`stg_customers` из Postgres, денормализует `channel` на каждый
заказ при загрузке (типичный OLAP-паттерн: JOIN один раз при вставке, а не
на каждый аналитический запрос). Идемпотентно — `TRUNCATE` перед вставкой,
как в transform-слое `etl-portfolio`.

**Честный замер (`clickhouse/compare_engines.py`).** Один и тот же запрос
("выручка по каналу/месяцу") тремя способами, медиана из 20 прогонов на
реальных 1985 заказах:

| Способ | Медианное время |
|---|---|
| Postgres VIEW-стиль (пересчёт с нуля) | **2.1 ms** |
| ClickHouse, агрегация сырых событий с нуля | 13.2 ms |
| ClickHouse, чтение инкрементальной витрины (`-Merge`) | 5.3 ms |

**На этом объёме данных Postgres быстрее ClickHouse во всех трёх вариантах**
— в том числе быстрее собственной инкрементальной витрины ClickHouse. Это не
баг замера, а честный и объяснимый результат: колоночный движок платит
фиксированную цену за HTTP-запрос и за то, что 60 итоговых строк — это
меньше, чем накладные расходы движка, спроектированного сканировать
миллиарды. Реальное преимущество `MergeTree`/колоночного хранения и
инкрементальных агрегатов проявляется на объёмах, которые физически не
влезают в план Postgres целиком в память или требуют полного
пересканирования при каждом запросе — не на 2000 строках синтетического
пет-проекта. Демонстрируется здесь архитектура (как правильно спроектировать
`ORDER BY`/`PARTITION BY`/инкрементальный rollup), а не измеренный выигрыш —
и это указано прямо, а не скрыто за круглыми цифрами.

## A/B-тестирование (`ab_test/`)

Симулированный, но методологически настоящий A/B-тест: email-кампания
реактивации на реальном пуле из 200 клиентов (`stg_customers`),
случайное разбиение 50/50 (`ab_test/generate_experiment.py`, зерно
зафиксировано для воспроизводимости) на control (обычная тема письма) и
treatment (персонализированная), метрика — конверсия в заказ в течение
14 дней после отправки.

Порядок проверок в `ab_test/analyze_experiment.py` — намеренный:

1. **Sample Ratio Mismatch (SRM)** — chi-square goodness-of-fit
   наблюдаемых размеров групп против ожидаемого сплита, порог `p < 0.01`
   (не 0.05 — ложные срабатывания здесь особенно дорого стоят: SRM
   ловит поломанную рандомизацию/трекинг, из-за которой любой
   результат теста дальше бессмысленно интерпретировать). Проверяется
   **до** теста значимости, а не после.
2. **Two-proportion z-test** — статистическая значимость разницы в
   конверсии между группами, плюс 95% доверительный интервал для самой
   разницы (доверительный интервал считается по непуловой стандартной
   ошибке — это стандартная практика, а не два случайных разных метода).
3. **Power-анализ** — сколько клиентов на группу нужно было бы для 80%
   мощности при наблюдаемом эффекте, и какую мощность реально даёт
   имеющийся размер выборки.

### Честный результат прогона

```
Control:   n=100, conversions=6,  rate=6.0%
Treatment: n=100, conversions=11, rate=11.0%

SRM check: p=1.000 (OK, randomization looks clean)
Two-proportion z-test: p=0.205 (не значимо на уровне 0.05)
Difference: +5.0%, 95% CI [-2.7%, +12.7%] — интервал пересекает ноль

Power-анализ: при этих ставках конверсии для 80% мощности нужно ~485
клиентов на группу — почти в 5 раз больше, чем есть в датасете (100).
Текущая выборка даёт только ~24% мощности.

Verdict: NO SHIP (inconclusive)
```

Это специально не подогнанный под "красивый" результат прогон:
"истинный" эффект (baked into `generate_experiment.py`, скрыт от самого
теста) — 8% против 12% конверсии, разница правдоподобная для реальной
маркетинговой рассылки. На выборке в 200 клиентов её статистически
не поймать — доверительный интервал пересекает ноль, а p-value больше
0.05. Это не баг генератора и не слабость теста — это ровно то, что
показывает power-анализ: при таком эффекте и такой выборке мощность
всего ~24%, то есть тест **в среднем** пропустил бы реальный эффект в
3 случаях из 4. Отсутствие значимости здесь честно не равно отсутствию
эффекта, и в отчёте это сказано прямо, а не спрятано за одной цифрой
p-value — ровно то же самое дисциплинированное правило, что и у churn-
модели с AUC≈0.5 выше.

9 unit-тестов (`tests/test_ab_test.py`) проверяют сами статистические
функции против руками просчитанных и scipy-сверенных значений — включая
то, что SRM-проверка ловит нарочно перекошенный сплит (70/130 вместо
100/100) и что мощность на required-размере выборки действительно
выходит на целевые 80%.

### Monte-Carlo валидация power-анализа (`ab_test/simulate_power_validation.py`)

Возникает соблазн: раз честный тест на 200 клиентах не значим, давайте
просто сгенерируем эксперимент побольше и получим красивый p-value. Это
была бы подгонка результата (p-hacking) — а на required-размере выборки
(рассчитанном на 80% мощности) один прогон и без того даёт значимость
только в ~80% случаев, не гарантированно.

Вместо этого — независимая проверка самой формулы `achieved_power`:
1000 симуляций эксперимента при тех же истинных ставках (8% против 12%)
на текущем размере выборки (n=100/группу) и на расчётном (n=880/группу,
для 80% мощности), с подсчётом, в какой доле симуляций тест реально
находит значимость.

```
True rates: control=8%, treatment=12%
Required n per arm for 80% power (formula): 880

current real sample (n=100/arm):
  Theoretical power (formula): 15.4%
  Empirical power (1000 simulations): 16.4%

required sample (n=880/arm):
  Theoretical power (formula): 79.9%
  Empirical power (1000 simulations): 80.1%
```

Теоретическая и эмпирическая мощность совпадают с точностью до случайного
шума симуляции в обоих случаях — формула `achieved_power` подтверждена
независимым методом, а не просто принята на веру.

Честная деталь, которую стоит проговорить отдельно: здесь 880 клиентов
на группу для 80% мощности, а в честном прогоне выше — "нужно ~485". Это
не противоречие, а два разных входа в одну и ту же формулу: 485 посчитан
по **наблюдаемым** ставкам конкретного прогона (6% и 11% — эффект
случайно показался больше истинного), а 880 — по **истинным** заданным
ставкам генератора (8% и 12%). Power, посчитанный постфактум по
наблюдаемым пропорциям малой выборки, сам по себе шумная оценка — отсюда
и разница. Планировать размер выборки по постфактум-мощности из уже
собранных данных — известная статистическая ловушка, а не техника,
которую стоит использовать всерьёз.
